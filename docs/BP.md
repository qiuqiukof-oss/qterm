# Hesi（合思）商业计划书 · BP 模板（融资用）

> 版本：v0.1.0-draft · 数据基准日：2026-07 · 主要依据：`Hesi-产品打磨实施方案.md` 现状盘点 + 代码库实地勘察
> 说明：本文为**融资对外模板**，所有"已实现"均来自代码核查；数字部分给出**假设与公式**，落地时由真实运营数据回填，严禁直接当作既定业绩引用。

---

## 1. 产品一句话定位

**Hesi（合思）** —— 在浏览器里运行任何 CLI / 终端，并让多个 AI 编程 Agent 以"圆桌讨论"方式协作，再通过 60+ 中国企业与开发工具连接器，把命令行能力接入真实的业务系统。

> 更短版本：**"浏览器里的终端中枢 + 多 Agent 圆桌 + 中国企业连接器生态"**。

---

## 2. 市场规模与机会

### 2.1 三个叠加的市场

| 切片 | 市场描述 | 与 Hesi 的对应能力（已实现） |
|------|----------|------------------------------|
| **浏览器终端 / Web IDE 内终端** | 开发者越来越习惯"不装客户端"地访问算力与终端（典型：各类 Cloud IDE、SSH Web 面板）。 | `node-pty` + `xterm.js` + WebSocket 多会话终端（已实现，`ws-handler.js` / `public/`） |
| **AI Agent 协作 / 多模型编排** | 单 Agent 天花见顶，"多个 Agent 互相质疑、收敛方案"成为新范式（圆桌、辩论、评审）。 | 多 Agent 圆桌协作：AI 助手 ↔ opencode 等 CLI Agent 回合制 SSE（已实现，`routes/chat/discuss.js`） |
| **中国企业连接器生态** | 国内企业数字化高度依赖飞书 / 企业微信 / 腾讯文档 / 钉钉 / 腾讯云等，但缺少统一、可编排的入口。 | `vendor/connectors/` + `mcp/` 已内置 **60+ 连接器**（飞书、企业微信、腾讯文档、钉钉、Jira、GitHub、各腾讯内部系统等） |

### 2.2 为什么是现在

- **终端上云 + 浏览器原生能力成熟**：WebSocket / WASM / xterm.js 让"浏览器即终端"体验已可用，不再是噱头。
- **AI 编程 Agent 普及**：opencode / codex / claude / aider 等 CLI Agent 已成开发者日常，但"在浏览器里、多 Agent 同框讨论"仍是空白。
- **企业 connector 碎片化**：每个 SaaS 都有 API，但缺乏"在统一工作台里被终端 + Agent 调用"的轻量入口；Hesi 已抢先沉淀 60+ 连接器，形成**先发渠道资产**。

### 2.3 机会判断（克制表述）

我们不声称"重新定义操作系统"。真实、可防守的机会是：**成为"企业 CLI + Agent + 连接器"的轻量编排中枢（orchestration hub）**，先把开发者与开源社区做厚，再以企业治理能力（审计 / RBAC / SSO / 团队）变现。

## 3. 护城河叙事（聚焦"生态渠道 + 先发"，避免夸大）

> 叙事原则：用**已落地、可演示、可核查**的资产讲护城河，不讲"技术壁垒天下无敌""要做一个操作系统"这类融资易翻车的表述。

### 3.1 真实已落地的护城河资产

| 护城河维度 | 证据（代码位置） | 状态 |
|------------|------------------|------|
| **多 Agent 圆桌协作** | `routes/chat/discuss.js`（回合制 SSE，AI ↔ CLI Agent 交替发言，含 token 统计） | ✅ 已实现 |
| **Headless 执行引擎** | `lib/cli-headless.js`（`opencode run` 无 TTY 调用）+ `lib/terminal-clean.js` | ✅ 已实现 |
| **MCP 安全三件套** | `mcp/security/{auth,policy,audit}.js` + 统一审计总线 `lib/audit.js` | ✅ 已实现 |
| **命令策略 / 限流** | `mcp/security/policy.js`（默认 blocklist）+ `ws/pty-policy.js` + `rate-limiter.js` | ✅ 已实现 |
| **企业连接器生态** | `vendor/connectors/`（60+，含飞书/企微/腾讯文档/钉钉/腾讯内部系统） | ✅ 已实现 |
| **RBAC + 许可矩阵** | `lib/auth/rbac.js`（admin/user/viewer）+ `lib/license.js`（能力门控） | ✅ 已实现 |
| **离线便携（无 Electron）** | `tray/tray.js` + `tray.exe`（Node SEA 编译）+ 便携 `node/` | ✅ 已实现 |

### 3.2 护城河的正确讲法

- **生态渠道护城河**：60+ 连接器不是一朝一夕能凑齐的，且与企业内部的"*-woa"类连接器（腾讯系）具有强渠道绑定属性。先发者的 connector 清单、鉴权对接经验、企业落地案例，构成**渠道与数据飞轮**。
- **先发与产品形态护城河**：把"浏览器终端 + 多 Agent 圆桌 + 连接器"三者组合成单一产品形态的，目前公开领域稀少；先发意味着社区心智、GitHub Stars、开发者口碑的累积。
- **可治理的轻量内核护城河**：审计总线、RBAC、命令策略、离线文件夹——这些"卖企业"的底座已就绪，而友商多为纯个人工具、缺治理层。

### 3.3 不夸张、可验证的边界

- 我们**不**自研大模型，模型成本受外部 API 影响（但"私有部署 + 自带 Key"反成商业卖点，把成本转嫁客户）。
- 我们**不**做重客户端（无 Electron），这是刻意的轻量取舍，不是能力缺失。
- 护城河是"生态 + 先发 + 治理底座"，**不是**"技术黑箱不可复制"。

## 4. 估值逻辑（假设 + 公式，非凭空数字）

> 本节的金额均为**建模假设占位符**，用 `{{ }}` 标注，需以真实数据回填。给出可讲清的口径，而非拍脑袋估值。

### 4.1 收入模型（三驾马车）

1. **SaaS 席位费（ARR 主力）**
   - 商业版按**席位 / 月**收费：`ARR = 付费席位数 × 每席 ARPU × 12`
   - 假设：`每席 ARPU = {{ 80–300 元/月 }}`，从开发者个人版逐步上探到企业团队版。
2. **连接器 / 生态抽成**
   - 对通过 Hesi 调用的第三方商业连接器按调用量或绑定License 抽成：`连接器收入 = Σ(单连接器调用量 × 单价 × 抽成率)`
   - 假设：`抽成率 = {{ 10%–30% }}`，优先从企业级连接器（企微/飞书/腾讯文档 SaaS 对接）起步。
3. **私有部署 / 商业 License 一次性 + 年维**
   - `私有部署收入 = 部署 License 费 + 年维护费(≈20%–25%)`
   - 对"数据不出内网"的政企/金融客户尤其适用（契合本项目的离线文件夹 + 审计能力）。

### 4.2 估值锚（用可比 SaaS 口径倒推）

```
估值 ≈ ARR × 行业乘数(ARR Multiple)
```
- 开发者工具 / DevTools SaaS 的合理 ARR 乘数区间约 `{{ 6×–12× }}`（公开可比参照：DevTools / 协作工具上市与一级市场区间，落地时引用具体可比公司）。
- 例（**仅作公式演示，非承诺**）：若 12 个月后达成 `ARR = {{ 付费席位 500 × ¥150 × 12 = ¥90万/月 ≈ ¥1080万/年 }}`，则 `估值 ≈ ¥1080万 × {{ 8× }} = {{ ¥8640万 }}`（占位，待真实数据）。

### 4.3 关键运营指标（融资时必须能拿出）

| 指标 | 公式 / 口径 | 当前状态 |
|------|------------|----------|
| DAU / MAU | 日/月活跃终端会话账号数 | 规划中（需遥测 `lib/telemetry.js`） |
| 付费 / 试点标杆数 | 已签约或 POC 企业客户数 | 规划中（Phase 4） |
| 连接器 Top N | 调用量前 N 的连接器及占比 | 规划中（需指标端点） |
| 圆桌采用率 | 启用多 Agent 讨论的会话占比 | 规划中（需遥测） |

> 重要：当前版本**零遥测**，因此上述指标在补齐 `lib/telemetry.js`（默认关闭、可开启）前**无法对外引用**。这是融资前必须补的短板（见实施方案工作流 C）。

## 7. 联创计划（明确需要 1 名商业 / BD 联创）

> 这是真实风险点：当前项目由单人推进技术，企业销售与生态对接**无法靠技术单人完成**。

### 7.1 为什么必须找联创

- 企业客户（尤其政企 / 金融）的采购链路长、需线下 BD、需等保/合规对接——技术创始人无法并行。
- 连接器生态的"*-woa"类腾讯内部系统对接，需要**有腾讯/企微/飞书渠道资源**的人推进。
- 融资叙事需要"技术 + 商业"双核，单人故事在 DD（尽职调查）中会被视为短板。

### 7.2 联创画像（招募条件）

| 维度 | 要求 |
|------|------|
| 角色 | **商业 / BD / 增长联创（Co-founder, Business）** |
| 经验 | 3–5 年以上 ToB SaaS / 开发者工具 / 企业服务销售或生态合作经验 |
| 资源 | 有腾讯/企微/飞书或云厂商渠道优先；有企业客户 POC 推进经验 |
| 产出 | 6 个月内拿到 2–3 家标杆 POC、跑通 PMF 阈值、主导种子轮 BD |

### 7.3 股权与分工（建议，占位）

- 技术创始人：产品 / 研发 / 开源社区。
- 商业联创：销售 / BD / 生态 / 融资辅助，对应 `{{ 30%–45% }}` 联创股（按到位里程碑释放）。
- 明确**里程碑对赌**：联创股随"标杆客户 / 募资"达成分批归属（vesting）。

## 8. 风险与缓解（融资必答题）

| 风险 | 缓解 |
|------|------|
| 单人资源瓶颈 | 募资首要补**商业联创**；Phase 3–4 前必须到位 |
| 开源转付费反弹 | 免费版保留核心价值（多 Agent、离线、开源）；仅对企业治理能力（审计/RBAC/SSO/团队/私有部署）收费 |
| 模型成本不可控 | 商业版"私有部署 + 自带 Key"把成本转嫁客户，反成卖点 |
| 遥测缺失无料可讲 | 募资前优先补 `lib/telemetry.js`（默认关闭），否则估值无支撑 |
| 公网暴露安全隐患 | 默认回环 + 单令牌 + 命令策略已就绪；提供 `SECURE_DEPLOYMENT` 指南 |
